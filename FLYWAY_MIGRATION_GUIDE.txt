1) What is Flyway and why use it
================================================================================

BEGINNER-FRIENDLY OVERVIEW (READ THIS FIRST)
=============================================

So far you learned that Flyway helps you version your database schema using migration files.
Before we dive into technical steps, here's a plain-language overview and a concrete explanation
why `spring.jpa.hibernate.ddl-auto=update` is OK for quick experiments but dangerous for real projects.

Short story:
 - On your laptop you can let Hibernate auto-update the schema. It "just works".
 - But when many developers and environments exist, automatic updates cause conflicts, silent data loss,
   and deployment instability.

Why NOT to use `spring.jpa.hibernate.ddl-auto=update` in production
-----------------------------------------------------------------
- It's uncontrolled: Hibernate will change the database schema automatically based on your entity classes.
  You don't get a versioned, reviewable script for the change.
- It's not repeatable: two developers may produce different schema states on their machines.
- It can silently drop or alter columns/data when entities change (risking data loss).
- Hard to audit: no history of "who changed what and when" stored in the repo.
- Not safe for rollbacks: reverting code may not revert the database; migrations make rollbacks predictable.

Concrete examples of problems with `ddl-auto=update`:
- Developer A adds column `age` to User; Developer B renames `age` to `years` and pushes later.
  Depending on timings, the DB may end in a broken state or lose data.
- An entity change accidentally renames a column or changes type and Hibernate executes ALTER TABLE that
  causes data truncation or errors in production.

What should you use instead?
---------------------------
Use a migration tool like Flyway (or Liquibase) to manage schema changes explicitly.
Migrations are SQL (or Java) scripts checked into version control so:
 - Every schema change is explicit, reviewed, and attributed to a commit/author.
 - Environments (dev/staging/prod) apply the same scripts in the same order.
 - You can inspect the history via the `flyway_schema_history` table.

Recommended `application.properties` settings when using Flyway
----------------------------------------------------------------
# Let Flyway manage schema; do NOT let Hibernate change schema automatically
spring.jpa.hibernate.ddl-auto=none

# Flyway settings
spring.flyway.enabled=true
spring.flyway.baseline-on-migrate=true
spring.flyway.locations=classpath:db/migration

Notes:
- `ddl-auto=none` (or simply remove the property) ensures Hibernate won't try to change schema.
- `baseline-on-migrate=true` is helpful when introducing Flyway to an existing DB: it records the
  current schema version (baseline) so Flyway can apply new migrations safely.

Quick transition steps (how to move from `ddl-auto=update` to Flyway safely)
--------------------------------------------------------------------------
1) Prepare: commit all code changes and ensure your working database is in a consistent state.
2) Create a baseline migration that represents the current schema (V1__baseline.sql).
   - Option A: Generate DDL from the existing database (mysqldump --no-data) and save as V1.
   - Option B: Write a V1 migration that creates the schema matching current state.
3) Add Flyway dependency to `build.gradle` and create `src/main/resources/db/migration`.
4) Set `spring.jpa.hibernate.ddl-auto=none` in your application.properties (or for production profile)
   and enable Flyway: `spring.flyway.baseline-on-migrate=true`.
5) Start the application once so Flyway creates `flyway_schema_history` (or run `./gradlew flywayBaseline`).
6) From this point on, any schema change must be implemented via a new migration file (V2__, V3__, ...)
   and committed to version control.

Simple example: Baseline workflow using MySQL dump
 - On a machine where schema matches production, run:
   mysqldump --no-data -u root -p Uber_DB_Local > baseline_schema.sql
 - Save `baseline_schema.sql` as `V1__baseline.sql` into `src/main/resources/db/migration`.
 - Set `spring.flyway.baseline-on-migrate=true` and start app (or run `./gradlew flywayBaseline`).

Now Flyway knows the DB's baseline and will apply future migrations from V2 onwards.


- Flyway is a database migration tool that keeps your database schema versioned and
  synchronized with your application code. Each change is represented as a migration
  file and Flyway ensures they are applied in a controlled order.

- Benefits:
  - Reliable, repeatable schema changes across environments (dev/stage/prod)
  - Simple SQL-based workflow (or Java-based migrations when necessary)
  - Integrates with Spring Boot automatically if on the classpath
  - Provides commands to inspect and repair migration state

When to use:
- Any application that requires schema evolution over time (most applications)
- Especially important when multiple developers/CI pipelines modify schema

2) How Flyway integrates with Spring Boot
================================================================================

- Spring Boot detects Flyway on the classpath and runs migrations automatically on
  application startup before Hibernate (if enabled) attempts to validate or update schema.
- This is convenient: simply adding the dependency will trigger migrations when you run
  the app with the same datasource configuration used by your application.

Important Spring Boot properties (defaults):
- spring.flyway.enabled=true  # run Flyway at startup
- spring.flyway.locations=classpath:db/migration  # default location for migrations
- spring.flyway.baseline-on-migrate=false

Note: If you use Hibernate's `spring.jpa.hibernate.ddl-auto=update` simultaneously, avoid
conflicts. For production, prefer Flyway-managed schema and set ddl-auto to `none` or remove it.

3) Add Flyway to your Gradle project
================================================================================

Option A - Add Flyway dependency only (migrations run by Spring Boot on app startup)

Edit `build.gradle` (add under dependencies):

dependencies {
    // ...existing dependencies...
    implementation 'org.flywaydb:flyway-core:9.16.0'  // choose a current stable version
}

Option B - Use Flyway Gradle plugin (gives tasks like flywayMigrate, flywayInfo)

// Top of build.gradle (plugins)
plugins {
    id 'org.springframework.boot' version '3.1.0'
    id 'io.spring.dependency-management' version '1.1.0'
    id 'java'
    id 'org.flywaydb.flyway' version '9.16.0' // optional: provides gradle tasks
}

// Configure plugin (example)
flyway {
    url = 'jdbc:mysql://localhost:3306/Uber_DB_Local'
    user = 'root'
    password = 'root'
    schemas = ['public']
    locations = ['filesystem:src/main/resources/db/migration']
}

Note: If you configure Flyway via Spring Boot properties, you can avoid duplicating connection
info in Gradle. The plugin is useful for running migrations independently from the Spring Boot app.

4) Configure Flyway in `application.properties`
================================================================================

Your project already has `src/main/resources/application.properties` with DB connection.
Add Flyway settings there (or application-dev.properties for dev env):

# datasource (already present in your project)
spring.datasource.url=jdbc:mysql://localhost:3306/Uber_DB_Local
spring.datasource.username=root
spring.datasource.password=root

# Flyway settings
spring.flyway.enabled=true
spring.flyway.locations=classpath:db/migration
spring.flyway.baseline-on-migrate=true   # use when you add Flyway to an existing DB
spring.flyway.placeholder-replacement=true

# Optional: control automatic execution (set to false in some CI scenarios)
# spring.flyway.enabled=false

Important: `baseline-on-migrate=true` is very useful when integrating Flyway into an
existing database that already has schema (it marks existing state as baseline so
Flyway can start applying new migrations from V1 or higher). Use carefully.

================================================================================
5) Migration file naming, versioning and locations
================================================================================

Default location:
- `src/main/resources/db/migration` (on classpath)

SQL migration file naming rules (standard):
- Versioned migrations: V{version}__{description}.sql
  e.g. V1__create_tables.sql, V2__add_sample_data.sql
- Repeatable migrations: R__{description}.sql
  e.g. R__refresh_views.sql (re-run when checksum changes)

Examples:
- src/main/resources/db/migration/V1__create_schema.sql
- src/main/resources/db/migration/V2__insert_seed_data.sql
- src/main/resources/db/migration/R__refresh_search_view.sql

Version format examples:
- V1
- V1_1
- V2026_05_19

Important rules:
- Avoid gaps and conflicts in version numbers (Flyway orders migrations by version)
- Each migration must be immutable once applied in production (prefer new migration over editing applied one)

6) Example migrations (full SQL) for your codebase
================================================================================

Below are complete example SQL migrations you can drop into
`src/main/resources/db/migration` for local development and learning.

DO NOT RUN against production without reviewing. These examples are for learning.

-- V1__create_schema.sql
-- Creates tables for Driver, Passenger, Review, Booking matching your entities

CREATE TABLE IF NOT EXISTS driver (
  id BIGINT NOT NULL AUTO_INCREMENT PRIMARY KEY,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  license_number VARCHAR(255) NOT NULL UNIQUE,
  name VARCHAR(255)
);

CREATE TABLE IF NOT EXISTS passenger (
  id BIGINT NOT NULL AUTO_INCREMENT PRIMARY KEY,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  name VARCHAR(255)
);

CREATE TABLE IF NOT EXISTS booking_review (
  id BIGINT NOT NULL AUTO_INCREMENT PRIMARY KEY,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  content TEXT,
  rating DOUBLE
);

CREATE TABLE IF NOT EXISTS booking (
  id BIGINT NOT NULL AUTO_INCREMENT PRIMARY KEY,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  booking_status VARCHAR(50),
  start_time TIMESTAMP NULL,
  end_time TIMESTAMP NULL,
  total_distance BIGINT,
  driver_id BIGINT,
  passenger_id BIGINT,
  review_id BIGINT,
  FOREIGN KEY (driver_id) REFERENCES driver(id),
  FOREIGN KEY (passenger_id) REFERENCES passenger(id),
  FOREIGN KEY (review_id) REFERENCES booking_review(id)
);

-- V2__insert_seed_data.sql
-- Insert sample data matching the examples used in guides

INSERT INTO driver (license_number, name) VALUES

- Table names and column names used above follow your built classes (e.g. booking_review table)

7) Java-based migrations (when SQL is insufficient)
================================================================================

- Flyway supports Java classes that implement `org.flywaydb.core.api.migration.JavaMigration`
or extend `BaseJavaMigration` for complex schema updates or data transformations.
- Place Java migrations on the classpath (e.g. in `src/main/java/db/migration` package)
- Example skeleton:

public class V3__migrate_complex_data extends BaseJavaMigration {
    @Override
    public void migrate(Context context) throws Exception {
        try (Statement st = context.getConnection().createStatement()) {
            st.execute("ALTER TABLE ...");
            // complex logic or data transformations
        }
    }
}

Use Java migrations for:
- Backfilling data with complex logic
- Calling external services during migration (rare)

8) Running migrations
================================================================================

Option 1 - Run automatically with Spring Boot
- When Flyway is on the classpath and `spring.flyway.enabled=true`, running `./gradlew bootRun` or
  running the Spring Boot app will run Flyway migrations before Hibernate initializes.

Option 2 - Run with Gradle plugin tasks (if plugin applied)
- ./gradlew flywayMigrate  # runs migrations
- ./gradlew flywayInfo     # shows migration state
- ./gradlew flywayValidate # validates checksums and applied versions

Option 3 - Use Flyway CLI
- Download Flyway CLI and configure `flyway.conf` or use command line args
- flyway -url="jdbc:mysql://localhost:3306/Uber_DB_Local" -user=root -password=root migrate

Testing locally:
1) Add migration SQL files to `src/main/resources/db/migration`
2) Run `./gradlew bootRun` or `./gradlew flywayMigrate`
3) Check Flyway schema history table: `flyway_schema_history` (created by Flyway)

================================================================================
9) Useful Flyway commands and their meaning
================================================================================

- migrate: Apply pending migrations (every startup with Flyway on classpath, or flywayMigrate)
- info: Show current migration state and pending migrations (useful to inspect)
- validate: Compare applied migrations with available migrations and checksums
- repair: Repair the Flyway metadata table (useful when manual edits cause checksum mismatch)
- baseline: Mark current schema as a baseline version (useful when introducing Flyway to an existing DB)
- clean: Drop all objects in the configured schemas (DANGEROUS - only in dev/test)

Examples:
- ./gradlew flywayInfo
- ./gradlew flywayValidate
- ./gradlew flywayRepair
- flyway -url=... -user=... -password=... info

================================================================================
10) Advanced topics
================================================================================

Placeholders
------------
- Flyway supports placeholders that can be replaced in SQL migration files.
- Example in application.properties:
  spring.flyway.placeholders.schema_name=public
- Use in SQL as ${schema_name}

Callbacks
---------
- You can add callbacks in `src/main/resources/db/migration` named like:
  - beforeMigrate.sql, afterMigrate.sql
  - beforeEachMigrate.sql, afterEachMigrate.sql
- Use callbacks for environment-specific tasks (creating DB users, logging, integration with monitoring)

Repeatable Migrations (R__)
---------------------------
- Good for views or stored procedures that you want to refresh when definition changes.
- Flyway will re-run repeatable migrations whenever checksum changes.

Undo (deprecated and limited)
-----------------------------
- Flyway used to support undo migrations in some versions; modern recommended practice is forward-only migrations
- Keep migration history immutable and write new migrations to undo changes instead of editing old ones

Transactional vs Non-transactional
----------------------------------
- Some database engines (e.g. MySQL with MyISAM) have non-transactional DDL. In that case, Flyway cannot
  wrap migrations in transactions. Prefer InnoDB for transactional behavior.
- Flyway will attempt to run each migration in a transaction if the DB supports it.

11) Best practices and team workflow
================================================================================

1) One migration per change
   - Keep migrations small and focused
2) Never edit a migration that's already been applied in production
   - If you must, create a new migration to fix or modify
3) Use semantic versioning for migration versions if helpful (V1, V2, V2_1, V3)
4) Keep migrations under version control alongside code
5) Use `baseline-on-migrate=true` when adopting Flyway on an existing DB
6) Use CI to run Flyway validate/info during builds to catch issues early
7) Keep destructive operations (DROP) behind feature flags or separate deploy steps
8) Document long running migrations and inform DBAs
9) Consider maintenance windows for heavy migrations (large data backfills)

12) Troubleshooting common errors
================================================================================

Error: Flyway schema history table not found / baseline needed
- Solution: If DB already has schema, set `spring.flyway.baseline-on-migrate=true` and set baselineVersion if needed.

Error: validate failed - checksum mismatch
- Cause: Someone edited an applied migration file
- Solution: If change is intended, add a new migration that corrects the issue. To fix local environment quickly, run `flyway repair` (but do not use repair to hide production inconsistencies lightly)

Error: Migration failed mid-way (partial failure)
- Solution: Fix SQL then use `flyway repair` to remove failed entry and re-run migrate (or if safe, rollback DB to previous state)

Error: ClassNotFound for Java migration
- Solution: Ensure Java migration class is on the classpath and follows Flyway package/class naming rules and extends `BaseJavaMigration`.

Error: Flyway not running on startup
- Solution: Check `spring.flyway.enabled=true` and that `org.flywaydb:flyway-core` is on classpath. Also ensure properties are under the active profile's properties file.

13) Example developer workflow (local)
================================================================================

1) Create migration file: `V3__add_new_column_to_booking.sql` in `src/main/resources/db/migration`
2) Commit migration to Git
3) Run locally: `./gradlew flywayMigrate` or `./gradlew bootRun`
4) Run tests, ensure application behavior is correct
5) Merge to feature branch, CI runs `flywayValidate` to ensure consistency
6) Deploy to staging; Flyway runs during application startup and migrates automatically
7) Deploy to production; migrations run as part of application startup or CI-run flyway tasks
